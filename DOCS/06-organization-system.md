## 목적

### 왜 바꾸는가?

현재 방식: **선생님이 학생을 직접 등록**
- 선생님이 학생 데이터 "소유"
- 학생은 수동적 객체
- 학원 이동 시 데이터 단절

신규 방식: **학생이 자기 계정 소유 + 기관에 소속**
- 학생이 데이터 주체
- 여러 기관 동시 소속 가능
- 학원 옮겨도 성적 기록 유지
- 학부모가 자녀 계정 연결 가능

### 기대 효과

| 항목 | 기존 | 신규 |
|------|------|------|
| 데이터 소유 | 선생님 | 학생 본인 |
| 학원 이동 | 데이터 단절 | 기록 유지 |
| 복수 학원 | 불가능 | 가능 |
| 학부모 연결 | 선생님이 입력 | 학생 계정에 연결 |
| 장기 기록 | 선생님 탈퇴 시 소실 | 영구 보존 |

## 기관 종류 (6가지)

| 코드 | 한글명 | 설명 |
|------|--------|------|
| `academy` | 학원 | 사업자등록된 정규 학원 |
| `institute` | 교습소 | 소규모 교습소 |
| `study_room` | 공부방 | 동네 공부방 |
| `group_tutoring` | 그룹과외 | 2인 이상 그룹 과외 |
| `private_tutor` | 개인과외 | 1:1 과외 (1인 기관) |
| `school` | 학교 | 정규 학교 |

### 개인과외 특이사항
- 선생님 1명 = 기관 1개
- 기관 생성 시 자동으로 owner 지정
- 사업자등록증 대신 신분증 확인

---

## 핵심 원칙

1. **학생 = 여러 기관 동시 소속 가능**
   - 학원 + 개인과외 동시 가능
   - 각 기관에서 별도 관리

2. **학생이 직접 신청**
   - 학생 본인이 가입 후 기관 검색
   - "가입 신청" 버튼으로 요청
   - 기관에서 승인 후 소속

3. **승인 권한 = 원장 또는 선생님**
   - 기관 관리자가 승인/거절 결정
   - 승인 후 담당 선생님 배정

## 승인 플로우

### 기관 생성 플로우

```
[원장/강사]                    [운영자(봉쌤)]
    │                              │
    │  1. 기관 생성 신청            │
    │  + 사업자등록증 첨부          │
    │─────────────────────────────>│
    │                              │
    │                   2. 서류 검토│
    │                   (사업자번호 │
    │                    실제 확인) │
    │                              │
    │  3-A. 승인                    │
    │<─────────────────────────────│
    │  → 기관 활성화               │
    │  → 원장 권한 부여             │
    │                              │
    │  3-B. 거절 (사유 안내)        │
    │<─────────────────────────────│
```

### 학생 가입 플로우

```
[학생]                    [원장/선생님]
    │                          │
    │  1. MathIQ 회원가입       │
    │  (학생 역할 선택)         │
    │                          │
    │  2. 기관 검색             │
    │  (학원명, 지역)           │
    │                          │
    │  3. "가입 신청"           │
    │─────────────────────────>│
    │                          │
    │              4. 신청 검토 │
    │              (실제 수강생 │
    │               확인)      │
    │                          │
    │  5-A. 승인               │
    │<─────────────────────────│
    │  → 기관 소속 등록         │
    │  → 담당 선생님 배정       │
    │                          │
    │  5-B. 거절               │
    │<─────────────────────────│
```

## DB 스키마 설계

### organizations (기관)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| name | TEXT | 기관명 |
| type | TEXT | academy/institute/study_room/group_tutoring/private_tutor/school |
| address | TEXT | 주소 |
| phone | TEXT | 대표 연락처 |
| owner_id | UUID | 원장 user_id (FK → profiles) |
| status | TEXT | pending/approved/rejected |
| business_license_url | TEXT | 사업자등록증 이미지 URL |
| created_at | TIMESTAMPTZ | 신청 시간 |
| approved_at | TIMESTAMPTZ | 승인 시간 |

### organization_members (기관 소속 선생님)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| organization_id | UUID | FK → organizations |
| user_id | UUID | FK → profiles |
| role | TEXT | owner/teacher |
| joined_at | TIMESTAMPTZ | 가입 시간 |

### organization_student_requests (학생 가입 신청)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| organization_id | UUID | FK → organizations |
| student_id | UUID | FK → profiles |
| status | TEXT | pending/approved/rejected |
| message | TEXT | 신청 메시지 (선택) |
| requested_at | TIMESTAMPTZ | 신청 시간 |
| responded_at | TIMESTAMPTZ | 응답 시간 |
| responded_by | UUID | 승인/거절한 사람 |

### organization_students (기관 소속 학생)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| organization_id | UUID | FK → organizations |
| student_id | UUID | FK → profiles |
| joined_at | TIMESTAMPTZ | 소속 시작 |
| left_at | TIMESTAMPTZ | 탈퇴 시점 (NULL = 현재 소속) |

### student_teacher_assignments (담당 선생님 배정)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| student_id | UUID | FK → profiles |
| teacher_id | UUID | FK → profiles |
| organization_id | UUID | FK → organizations |
| subject | TEXT | 담당 과목 |
| assigned_at | TIMESTAMPTZ | 배정 시간 |

## 기존 데이터 처리

### 현재 students 테이블 구조
```
students
├─ id
├─ user_id (선생님 소유)  ← 이 구조가 문제
├─ name, grade, ...
```

### 전환 전략: 점진적 마이그레이션

**Phase A: 병행 운영**
- 기존 students 테이블 유지
- 신규 organization 테이블 추가
- 새 학생만 신규 방식으로 가입

**Phase B: 기존 학생 전환 유도**
- 기존 학생에게 "본인 계정 만들기" 안내
- 선생님이 학생 이메일 입력 → 초대 메일 발송
- 학생이 가입 후 기존 데이터 연결

**Phase C: 완전 전환**
- 일정 기간 후 기존 방식 deprecated
- 미전환 학생은 읽기 전용 유지

## 미해결 고민

### 1. 학교 등록은 어떻게?
- 사립학교: 사업자등록증 가능
- 공립학교: 재직증명서? 학교장 승인?
- 검토 필요: 학교는 Phase 3 이후로 미룰 수도

### 2. 학원 이동 시 데이터 권한
- 과거 학원에서 입력한 성적, 새 학원에서 볼 수 있나?
- **옵션 A**: 학생 본인만 조회 (새 학원은 못 봄)
- **옵션 B**: 학생이 공유 허용 설정
- **옵션 C**: 모든 담당 선생님 조회 가능

**현재 방향**: 옵션 A (학생 본인만)

### 3. 탈퇴 학생 데이터 보존
- 기관 탈퇴 시 데이터 삭제? 보존?
- 보존 시 얼마나? (1년? 3년? 영구?)
- 법적 요건 확인 필요

**현재 방향**: left_at만 기록, 데이터 영구 보존

### 4. 개인과외 선생님의 기관
- 개인과외도 "기관" 생성 필요한가?
- 아니면 기존처럼 직접 학생 등록 허용?
- 일관성 vs 편의성 트레이드오프

**현재 방향**: 개인과외도 1인 기관 생성 (일관성 우선)

## 주의사항

- **구현 전 봉쌤 재확인 필수**: DB 스키마 변경 전 반드시 승인
- **기존 데이터 손실 금지**: 마이그레이션 시 롤백 계획 필수
- **점진적 전환**: 빅뱅 전환 금지, 병행 운영 후 전환
- **RLS 설계 중요**: 학생 ↔ 기관 ↔ 선생님 권한 복잡함
